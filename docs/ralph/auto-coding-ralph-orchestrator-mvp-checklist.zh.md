# Ralph Orchestrator MVP 范围与验收清单

版本：v1.0 草案  
日期：2026-04-30  
文档语言：中文  

---

## 1. MVP 目标

MVP 目标不是一次性做完整自动开发平台，而是先验证一个可靠闭环：

```text
需求清楚
任务够细
上下文隔离
上下文可恢复
权限可控
执行可审计
验收有证据
报告能看懂
```

---

## 2. MVP 必须包含

1. 中文 PRD 管理。
2. brainstorm 边问边记录。
3. brainstorm 独立验收。
4. story 和 dev task 管理。
5. 任务颗粒度门禁。
6. OpenSpec-style `.ralph/specs/current` 和 `.ralph/specs/changes`。
7. AI 工作单元 schema。
8. `task_harness` schema。
9. harness 执行前门禁。
10. harness 执行中检查点。
11. harness 执行后门禁。
12. 状态机。
13. 状态持久化。
14. Claude Code 短任务调用。
15. 执行日志。
16. 阻塞队列。
17. 安全权限策略。
18. 单文件删除备份。
19. 测试结果记录。
20. 独立 review。
21. 用户路径验收。
22. Playwright 真实点击验收。
23. 多尺寸截图验收。
24. 边界状态检查清单。
25. review 问题转任务机制。
26. 中文研发报告。
27. `ToolAdapter` 抽象接口和至少一个 Claude Code 适配器。
28. LLM Provider 基础配置模型和至少一种 Provider 支持。
29. Issue 源抽象和本地 issue 文件读取。

---

## 3. MVP 暂不包含

1. 完整 Web dashboard。
2. 大规模并行。
3. 复杂权限 UI。
4. 团队多人协作。
5. 完整成本计费系统。
6. Codex、Aider、Cline 等额外工具适配器。
7. GitHub Issues 实时同步和 webhook。
8. LLM 驱动的 Issue 自动分类和自动修复。
9. 多 Provider 自动降级和负载均衡。

---

## 4. brainstorm 验收清单

brainstorm 合格后，必须能回答：

1. 这个产品给谁用。
2. 用户最常见的使用流程是什么。
3. 有哪些角色。
4. 每个角色能做什么。
5. 哪些功能第一版必须有。
6. 哪些功能暂不做。
7. 系统需要保存哪些数据。
8. 有哪些权限规则。
9. 有哪些业务规则。
10. 成功路径是什么。
11. 失败路径是什么。
12. 空状态和异常状态是什么。
13. 用户认为“真的可用”的标准是什么。
14. 第一版如何验收。
15. 还有哪些未确认假设。

不合格条件：

1. 目标用户不清楚。
2. 核心流程不清楚。
3. 角色权限不清楚。
4. 第一版范围不清楚。
5. 验收标准不清楚。
6. 缺少用户原话追溯。

---

## 5. PRD 验收清单

PRD 合格后，必须包含：

1. 背景。
2. 核心结论。
3. 用户目标。
4. 不做什么。
5. 产品定位。
6. 总体工作流。
7. 核心原则。
8. 核心需求。
9. MVP 范围。
10. 成功标准。
11. 关键风险。
12. 已确认设计前提。
13. 待实施确认项。

PRD 不得包含：

1. 未确认却被写成确定需求的 AI 猜测。
2. 与用户目标无关的炫技功能。
3. 无验收方式的模糊目标。

---

## 6. 任务颗粒度验收清单

每个 Claude Code 单次执行任务必须满足：

1. 只对应一个清晰开发动作。
2. 修改范围有限。
3. 有明确输入。
4. 有明确输出。
5. 有明确禁止修改范围。
6. 有依赖任务列表。
7. 有验收标准。
8. 有测试或检查方式。
9. 有回滚方式。
10. 有风险说明。
11. 理想执行时间在 10 到 30 分钟内。

不合格任务示例：

```text
完成登录系统。
优化整个前端体验。
重构后台管理模块。
```

合格任务示例：

```text
在 LoginForm 中增加 rememberMe 字段；
只修改 LoginForm 和对应测试；
不修改登录 API 合同；
提交时把 rememberMe 传给 auth service；
运行 LoginForm 测试。
```

---

## 7. 上下文隔离验收清单

每个关键工作单元必须满足：

1. 生成者和验收者不是同一个 agent 上下文。
2. 验收者读取结构化输入、输出、验收标准和证据。
3. 验收者不依赖生成者自述作为唯一依据。
4. 调度 agent 根据验收结果修改状态。
5. 验收不合格时创建返工任务。

必须独立验收的工作：

1. brainstorm。
2. PRD。
3. 任务拆解。
4. 验收标准。
5. 开发执行。
6. 测试有效性。
7. 最终报告。

---

## 8. 任务 Harness 验收清单

每个任务进入 `ready` 前，必须有 `task_harness`。

`task_harness` 必须明确：

1. 任务目标。
2. 允许读取的上下文。
3. 上下文大小和来源边界。
4. 允许修改范围。
5. 禁止修改范围。
6. 允许工具和命令。
7. 禁止工具和命令。
8. 执行前检查。
9. 执行中检查点。
10. 执行后验收门禁。
11. 必须保存的证据。
12. 失败重试规则。
13. 回滚方式。
14. 超时策略。
15. 停止和阻塞条件。
16. 独立验收者。

不合格条件：

1. 没有 `task_harness`。
2. harness 只有自然语言，没有结构化字段。
3. 没有明确禁止范围。
4. 没有证据要求。
5. 没有独立验收者。
6. 没有失败处理规则。
7. 没有执行后门禁。

---

## 9. 开发执行验收清单

执行 agent 完成后，必须提交：

1. 修改文件列表。
2. 新增文件列表。
3. 删除文件列表。
4. 是否越界修改。
5. 测试结果。
6. 证据文件。
7. 风险说明。
8. 对下游任务的影响。
9. harness 执行事件。
10. harness 违规情况。

任务不能仅凭以下内容验收：

1. “我已经完成。”
2. “看起来没问题。”
3. “测试应该能通过。”
4. “这是一个简单改动。”

---

## 10. 真实可用性验收清单

涉及前端、页面、表单、按钮、菜单、弹窗、流程的任务，必须尽量检查：

1. 页面能打开。
2. 关键按钮能点击。
3. 表单能输入和提交。
4. 成功路径能走通。
5. 失败路径有反馈。
6. 空状态正常。
7. 加载状态正常。
8. 权限不足状态正常。
9. 控制台没有新增明显错误。
10. 关键网络请求没有失败。
11. 桌面端截图正常。
12. 平板截图正常。
13. 手机截图正常。
14. 没有明显文字溢出、重叠、错位。

---

## 11. 安全验收清单

自动允许：

1. 读取项目文件。
2. 修改任务范围内的代码文件。
3. 创建普通代码文件。
4. 运行测试。
5. 运行 lint、typecheck、build。
6. 生成报告。

需要保护：

1. 删除文件。
2. 大规模移动文件。
3. 修改环境变量文件。
4. 修改密钥。
5. 修改系统目录。
6. 运行安装脚本。
7. 运行数据库删除或迁移。
8. 运行发布命令。
9. 运行不可逆命令。

单文件删除必须先备份到 `.ralph/backups/`。

---

## 12. 停止条件

出现以下情况，系统应停止自动推进或进入阻塞：

1. 连续失败超过阈值。
2. 阻塞项超过阈值。
3. 测试环境不可用。
4. 修改范围异常扩大。
5. 发现大规模删除。
6. 发现密钥、环境变量、数据库危险操作。
7. 验收标准缺失。
8. 上下文隔离无法满足。
9. 用户产品判断缺失。
10. `task_harness` 缺失。
11. `task_harness` 被违反。

---

## 13. MVP 完成定义

MVP 完成不是指系统已经能全自动完成任何项目。

MVP 完成是指：

1. 用户能用中文完成一次需求共创。
2. 系统能生成并冻结 PRD。
3. 系统能把需求拆成足够小的任务。
4. 系统能为任务生成 `task_harness`。
5. 系统能通过 harness 门禁调度 Claude Code 执行一个短任务。
6. 系统能保存执行状态和证据。
7. 系统能独立 review 执行结果。
8. 系统能运行测试或检查。
9. 系统能把 review 问题转成返工或阻塞项。
10. 系统能生成中文研发报告。
11. 中断后能从状态文件恢复。
12. `ToolAdapter` 接口已定义，Claude Code 适配器能正常调用并返回统一结果。
13. LLM Provider 配置模型已定义，至少支持一种 Provider 的连通性测试。
14. Issue 源抽象已定义，能从本地文件读取 issue 并写入状态系统。
